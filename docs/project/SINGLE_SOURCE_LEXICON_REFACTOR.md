# 从三套码表回到一个真源

2026 年 7 月，Yime 完成了一轮看似“回退”、实际上是收紧工程边界的重构：
系统不再把等长、变长、省键三套码表视为三个可以分别维护的事实来源，而只导入
一份等长码表。其余模式全部由程序按确定规则生成。

## 为什么要做

三套输入模式最初是同一套音元结构的不同书写形式。随着原型、PIME 集成和 ASCII
投影分别演进，三份运行词典逐渐具有了独立数据源的样子。一旦允许分别导入和修补，
同一个词的三种编码就可能不再互相对应。时间越长，人工补丁越多，差异越难判断究竟
来自原始数据、模式规则，还是键盘布局。

这轮重构采用一个简单约束：

> 外部数据如果有错，只应有一个外部错误源；模式关系如果有错，只应修改程序。

因此，今后替换一份等长 `.yaml` 就能替换词库事实；修改一处转换代码就能改变全部
词条的变长或省键关系。不再需要人工保持三份大文件同步。

## 新的数据链路

```text
唯一导入的 yime_full.dict.yaml
    -> 校验每个音节是否为完整四码
    -> 合并相邻同码、去除虚首音，生成变长码
    -> 根据音质组和调级省略中调，生成省键码
    -> 写出三套 Rime 运行词典
    -> 写出带源文件哈希和输出哈希的生成清单
```

三套 `dict.yaml` 仍然存在，因为 Rime 在输入期间需要高效查询预生成词典。它们的身份
已经改变：它们是可删除、可重复构建的运行产物，不再是可以独立编辑的源数据。模式
转换发生在导入和构建阶段，不会增加日常按键路径的计算负担。

拼音映射表也从四列收敛为两列：`pinyin_tone` 和 `full`。反查、用户词库生成等调用方
加载等长码后，在内存中使用同一个 `codemode` 包推导另外两种形式。这样，系统词典、
用户词典和反查不会再各自理解一套模式关系。

## 这次迁移发现了什么

迁移前先用新规则对旧数据做了全量核验：

- 1624 个拼音音节的旧变长列和省键列都能从等长列精确推导，差异为 0；
- 468,166 条等长系统词条可以完整生成三种模式；
- 旧变长词典与新规则在内容集合上相同；
- 旧省键词典中有 178,317 条仍沿用了变长码，没有应用正式的省键规则。

最后一项说明此前担心的分化已经真实发生，并非仅仅是架构上的可能性。本轮选择以
已经确认的正式省键规则重新生成这些词条，而不是继续保留历史产物中的不一致。

生成器还验证了固定点性质：把生成出的等长词典再次作为输入，得到的三套词典逐字节
一致。生成清单记录源 SHA-256、转换规则版本、布局版本、词条数和每个输出文件的
SHA-256，使安装包中的三套运行数据可以追溯到同一个来源。

## 如何使用

只生成项目内的三套运行词典：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File tools\import-yime-full-lexicon.ps1 `
  -Input C:\path\to\yime_full.dict.yaml
```

生成并部署到当前用户的 PIME/Rime 数据目录：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File tools\deploy-yime-rime-data.ps1 `
  -Input C:\path\to\yime_full.dict.yaml
```

导入脚本没有 `variable` 或 `shorthand` 模式开关。这是有意的：任何模式都不能再绕过
等长真源单独进入工程。

## 对键位调整意味着什么

这轮重构首先解除的是“三套运行码表互相牵制”。以后调整模式规则，不必同时改三份
词典；逐步调整码元与键位的映射，也不必迁就旧省键词典中已经固化的结果。

但目前还不能说音元语义与物理键位已经彻底解耦。正式规则中的音质分组仍以当前 ASCII
键符表达。下一阶段应把它进一步拆成：

```text
稳定的音元 ID 和属性
    -> 等长 / 变长 / 省键的语义转换
    -> 可替换的“音元 ID 到键位”布局投影
    -> Rime ASCII 编码
```

完成这一层后，局部换键、整体重排或试验新布局只需替换布局映射并重新生成运行词典，
不会改变词库，也不会无意中改变省键规则。

## 本轮刻意没有改变的部分

- 不合并三个 Rime schema；
- 不在每次按键时逐词转换；
- 不迁入整套 Python 拼音分析原型；
- 不改变候选注释统一显示等长 PUA 音元结构的做法；
- 不改变 native Rime 对候选分页的所有权；
- 不借这次数据重构调整语言栏菜单或宿主点击路径。

这使重构集中在“数据从哪里来、派生关系由谁负责”两个问题上，没有把已经稳定的 PIME
宿主集成重新卷入风险范围。

## 工程上的意义

这不是为了减少三个文件本身，而是为了减少三个事实来源。生成文件仍然可以很多，运行
模式仍然可以继续增加；只要它们都能从同一个真源和有版本的规则重复得到，复杂度就留在
可测试的程序内部，而不会扩散成人工维护的数据分叉。

从这个意义上说，这轮改造确实是一次回归：回到项目最初“只导入一套”的思路；同时也
是一次前进，因为这次单源关系已经由代码、测试、哈希清单和部署入口共同保证。
