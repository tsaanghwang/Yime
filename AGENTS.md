# Agent Constraints

These rules exist to prevent AI-assisted edits from destabilizing PIME/Rime host integration.

- Do not change native Rime candidate paging ownership without an explicit user request. `nativeBackend.UsesBackendCandidatePaging()` must keep returning `true`; real Rime sessions own their paging after activation and language-bar/menu clicks.
- Do not use Go-side candidate slicing as a shortcut to force the visible candidate count for native Rime sessions. Fix candidate count through Rime configuration/deploy/reload paths instead.
- Do not treat language-bar submenu depth as inherently unsafe. PIME may support nested menus, but changes to nested menu structure must be backed by a regression test that exercises the exact host click path.
- Before changing reverse lookup menu IDs, language-bar command IDs, candidate setting menu structure, or native backend click/activation behavior, add or update a regression test for the concrete failure path being protected. In particular, guard against clicking language-bar items such as "音元拼音" causing the host application to exit.
- The guard test `TestNativeBackendKeepsRimeOwnedCandidatePaging` is intentional. Do not weaken or remove it to make an unrelated candidate-window change pass.
- The Go-side `candidatePageSize` field must stay synchronized with Rime's `menu.page_size`. The sync mechanism works through `rimeState.PageSize`: `nativeBackend.State()` reads it from Rime, `applyStateToResponse` writes it back to `ime.candidatePageSize`, and `setCandidatePageSize` confirms the value after `reloadBackendForSchema`. Do not break this readback chain or introduce a separate Go-side page size that diverges from Rime's actual value.
- Do not assume Rime custom YAML keys are unquoted. Real user files may contain `"menu/page_size": 6`; any page-size read/write helper must support both quoted and unquoted `menu/page_size` keys or page-size changes will appear to "do nothing".
- When the candidate-count setting appears stuck, verify all three layers before changing logic: the language-bar command ID in `go_backend.log`, the actual files under `%APPDATA%\\PIME\\Rime`, and the installed runtime under `C:\Program Files (x86)\YIME`. Do not trust unit tests alone for this failure.
- Do not assume a source fix is live until you verify the installed `server.exe` and `rime_deployer.exe` were rebuilt, packaged, installed, and the running `PIMELauncher` and `server.exe` processes were restarted. Old installed binaries can make a correct source fix look ineffective.
- Do not assume language-bar submenu clicks always arrive as an integer top-level `req.ID` or `data.commandId`. Real host callbacks may report the clicked submenu item through `data.id` while the top-level button ID stays a string such as `reverse-lookup`; keep that compatibility covered by regression tests before refactoring command parsing.
- Do not "simplify" `commandIDFromRequest` by removing `data.id` fallback unless you have a host-level reproduction proving submenu clicks still reach `onCommand` correctly. Breaking submenu command decoding can make harmless reverse-lookup clicks look like host crashes or silent no-ops.
- When the reported bug is "language-bar click exits the host" or "menu click does nothing", verify the installed `C:\Program Files (x86)\YIME\go-backend\server.exe` timestamp changed and then reproduce once against the installed runtime before concluding the source fix worked. Source-only validation is not enough for TSF menu regressions.
- Do not simplify `Reinstall-PIME-Test.cmd` or `tools/templates/Reinstall-PIME-Test.cmd` back to uninstall-then-install only. The canonical reinstall flow must keep pre-flight (`dev-stop-pime.ps1 -Auto`), DLL-lock detection (exit code 2), and automatic in-place install fallback. `tools/refresh-dev-test-cmds.ps1` copies the template; edit the template, not an inline simplified generator.
- The Win32 PIMELauncher build requires the pinned `Rust_TOOLCHAIN=stable-i686-pc-windows-msvc` (set in the root `CMakeLists.txt`) plus Corrosion v0.6.1. Do not "fix" the build by unpinning the toolchain, removing `PIMELauncher/.cargo/config.toml`'s `build.target`, or downgrading Corrosion: with an x86_64 host toolchain, Corrosion leaks i686 target MSVC lib paths into host-side build-scripts (serde etc.), causing LNK4272 machine-type conflicts and 100+ unresolved externals. The prerequisite is `rustup toolchain install stable-i686-pc-windows-msvc`.
- When `git`/`cmake` FetchContent fails with "Failed to connect to github.com:443" on this machine, it is not a transient outage. The WinINET system proxy (`localhost:1081`) is enabled but command-line git/cmake do not read it; set `$env:HTTPS_PROXY = "http://127.0.0.1:1081"` (and `HTTP_PROXY`) before reconfiguring. Do not delete `build/CMakeFiles` casually — that invalidates FetchContent stamps and forces a re-clone of corrosion, which needs the proxy.
- When `Reinstall-PIME-Test.cmd` reports `PIMETextService.dll is still loaded (often explorer.exe)`, that is the designed in-place-install path, not a failure. A fully clean uninstall+reinstall only happens after a Windows reboot releases the DLL. Do not force-kill explorer or force-delete the DLL to "unblock" the script.
- CodeIntegrity Operational events 3033/3077 mentioning `PIMELauncher.exe` loading `go-backend\server.exe` are audit records from Smart App Control (this machine runs SAC in enforce mode) about the unsigned dev binary — not evidence of a YIME crash or a broken install. The log also contains unrelated noise (Bonjour `mdnsNSP.dll`, Keyman). Check the event message paths before attributing failures to YIME, and remember unsigned binaries may be blocked outright on other SAC/WDAC-enforced machines; public release requires trusted signing.
- Before concluding the Win32 (`build/`) tree is broken, check whether its artifacts were simply cleaned: `dev-install.ps1` hard-requires `build/PIMELauncher/PIMELauncher.exe` and `build/PIMETextService/Release/PIMETextService.dll` and aborts early if missing. The fix is to rebuild `build/` (Win32, with the pinned toolchain), not to weaken the install script's asserts.
